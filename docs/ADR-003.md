# ADR-003: Implementación de dos adaptadores de entrada (REST API y MVC con Thymeleaf)

## Estado
Aceptado

## Fecha
2026-05-15

## Contexto
El sistema debe atender dos formas de consumo:
1. integración programática mediante API HTTP (REST),
2. uso interactivo por navegador para demostración y operación básica (MVC + Thymeleaf).

Se busca reutilizar la misma lógica de negocio sin duplicarla.

## Problema Arquitectónico
Si cada interfaz implementa sus propias reglas:
- se duplica lógica,
- aparecen inconsistencias funcionales,
- y aumenta el costo de mantenimiento.

## Decisión
Definir dos adaptadores de entrada independientes:
1. REST Controller para endpoints JSON.
2. MVC Controller para flujo web servidor-renderizado.

Ambos adaptadores invocan los mismos puertos de entrada/casos de uso del núcleo de aplicación.

## Drivers Arquitectónicos (priorizados)
1. Reutilización de lógica de negocio
2. Coherencia funcional entre canales
3. Experiencia de uso para diferentes consumidores
4. Escalabilidad funcional del sistema

## Alternativas Consideradas
1. Solo API REST
2. Solo MVC
3. REST + MVC sobre los mismos casos de uso - ELEGIDA

## Consecuencias
### Positivas
- Un solo núcleo funcional para múltiples canales.
- Menor riesgo de divergencia funcional entre UI y API.
- Permite pruebas por canal (REST/MVC) y por núcleo (casos de uso).
- Cumple de forma clara con patrón Adapter solicitado.

### Negativas
- Mayor superficie de mantenimiento (controladores, DTOs, vistas).
- Posible duplicación de validaciones de entrada en ambos canales.
- Necesidad de políticas claras de errores para mantener consistencia de mensajes/códigos.

## Decisiones Derivadas
1. Los adaptadores solo transforman entrada/salida y delegan negocio.
2. El manejo de errores de negocio debe centralizarse en excepciones de dominio/aplicación.
3. Las validaciones de negocio no deben quedar en la capa web/rest, solo validaciones sintácticas o de formato.

## Riesgos Detectados y Mitigación
Riesgo 1: Duplicar reglas de negocio en controladores REST y MVC.  
Mitigación: validar reglas funcionales exclusivamente en casos de uso y dominio.

Riesgo 2: Inconsistencia entre respuestas REST y mensajes de la UI.  
Mitigación: definir catálogo de errores de negocio y mapearlo por canal.

Riesgo 3: Acoplar un adaptador de entrada con componentes de salida.  
Mitigación: mantener frontera de adaptadores; entrada consume puertos de entrada, no detalles de persistencia.

## Criterios de Cumplimiento del ADR
- REST y MVC llaman al mismo caso de uso para una operación equivalente.
- No existe lógica de negocio crítica en controladores.
- Los contratos de entrada/salida están separados por DTOs.
- Cambios en UI o API no requieren modificar reglas de negocio.

## Implicaciones a Futuro
- Se puede agregar un tercer canal (CLI, gRPC, mensajería) reutilizando los casos de uso existentes.
- Se favorece evolución incremental de la experiencia de usuario sin tocar el núcleo.
- Permite comparar y medir calidad por canal sin comprometer consistencia funcional.
